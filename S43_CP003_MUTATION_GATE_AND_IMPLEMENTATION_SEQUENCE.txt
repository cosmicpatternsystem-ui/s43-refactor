3. MUTATION GATE PRINCIPLE
------------------------------------------------------------

S43 must not move from blueprint to code mutation without a gate.

The gate exists to ensure that any implementation begins with:

- known files
- known baseline
- known safety state
- known rollback path
- known implementation order
- known forbidden actions
- known verification steps

No CP-003 code change should occur until the project has a pre-mutation inventory snapshot.

------------------------------------------------------------
4. REQUIRED PRE-MUTATION INVENTORY
------------------------------------------------------------

Before CP-003 code mutation is authorized, S43 must produce an inventory snapshot containing:

- list of relevant source files
- list of configuration files
- list of verifier or test files
- list of existing receipts and lock documents
- hash of critical files where practical
- detected entry points
- detected execution-related code
- detected exchange-related code
- detected wallet-related code
- detected AI-related code if any
- detected strategy or signal modules
- detected safety or risk modules

The inventory must be read-only.

The inventory must not modify code.

------------------------------------------------------------
5. FIRST IMPLEMENTATION STRATEGY
------------------------------------------------------------

The first CP-003 implementation must be structural and low-risk.

The preferred first implementation slice is:

CP003-A:
Architecture scaffold package and neutral interfaces.

CP003-A should introduce structure without changing trading behavior.

CP003-A may include:

- new scaffold directory
- module placeholder files
- neutral interface definitions
- safety boundary placeholder
- audit receipt placeholder
- autonomy mode placeholder
- AI supervisor placeholder
- execution approval placeholder
- architecture registry
- no-op behavior
- read-only defaults

CP003-A must not change live behavior.

8. FORBIDDEN DURING CP-003-A
------------------------------------------------------------

The first implementation slice must not:

- place orders
- cancel orders
- modify leverage
- move wallet funds
- enable withdrawals
- enable live trading
- grant AI execution authority
- change credential handling
- replace the baseline
- create a fourth codebase
- rewrite the project from zero
- change existing strategy behavior silently
- remove existing safety checks
- bypass verifiers

------------------------------------------------------------
9. ACCEPTABLE CP-003-A CODE CHANGES
------------------------------------------------------------

After explicit future mutation authorization, acceptable CP003-A code changes may include:

- adding new scaffold files
- adding new interface files
- adding new constants files
- adding new receipt schema files
- adding new read-only verifier files
- adding new documentation-linked markers
- adding no-op module registration
- adding deny-by-default policy objects

These changes must be isolated and reviewable.

10. VERIFICATION REQUIREMENTS
------------------------------------------------------------

Before CP003-A can be considered complete, verification must prove:

- Python compilation passes if Python source exists

- receipts can be generated or represented
- baseline identity remains preserved

------------------------------------------------------------
11. ROLLBACK REQUIREMENT
------------------------------------------------------------

Every CP-003 implementation step must be reversible.

The project must preserve:

- original baseline documents
- pre-mutation inventory
- file hashes where practical
- change receipts
- verifier output
- implementation notes

If a CP-003 change creates unclear behavior, it must be stopped before further mutation.
